# 评论系统设计

## 1. 背景与目标

当前博客是 Astro SSG 静态站点，文章详情统一由 `src/pages/blog/[...slug].astro` 渲染，并通过 `src/layouts/BlogPost.astro` 输出文章页面。评论系统需要作为独立后端接入，前端仅通过公开 API 读取和提交评论。

目标是实现一个可长期维护的个人博客评论系统，而不是完整社区系统。第一版优先保证可用、安全、可审核、可迁移。

## 2. 范围

### 2.1 第一版包含

- 游客填写昵称、邮箱和正文提交评论。
- 按评论目标 `postSlug` 展示已审核内容；目标可以是文章或站点功能页。
- 评论默认进入审核队列，管理员审核后公开。
- 管理员登录、查看待审核评论、批准、拒绝、删除评论。
- Cloudflare Turnstile 防刷。
- 基础限流、CORS、输入长度限制和纯文本渲染。
- 数据库迁移脚本和基础测试。

### 2.2 第一版不包含

- 用户注册登录系统。
- OAuth 登录。
- 富文本编辑器。
- 点赞、举报、订阅、站内通知。
- 无限层级楼中楼。
- 实时推送。
- 附件上传。

## 3. 推荐架构

```txt
Astro Blog
  └─ src/features/comments/ React 评论组件
       └─ HTTPS API
            └─ apps/comment-api/ Hono 服务
                 ├─ Drizzle ORM
                 ├─ PostgreSQL
                 ├─ Turnstile 校验
                 └─ 管理员 JWT/session
```

推荐技术选型：

| 层级 | 选型 | 说明 |
|---|---|---|
| API 框架 | Hono | TypeScript 友好，轻量，适合 serverless 和边缘运行时 |
| 数据库 | PostgreSQL | 评论数据结构化，迁移和备份能力成熟 |
| ORM | Drizzle ORM | 类型轻量，迁移文件清晰，适合小型 API |
| 防刷 | Cloudflare Turnstile | 免费，避免第一版自建验证码 |
| 部署 | Cloudflare Workers 或 Vercel Functions | 低流量博客现金成本低 |
| 前端 | Astro + React | 当前项目已启用 `@astrojs/react` |

## 5. 数据模型

### 5.1 comments

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | uuid | primary key | 评论 ID |
| post_slug | text | not null, indexed | 文章 slug，对应 Astro `post.id` |
| parent_id | uuid | nullable, indexed | 回复目标；第一版可先保留字段但不开放多级回复 |
| author_name | varchar(64) | not null | 昵称 |
| author_email_hash | char(64) | nullable | 邮箱 SHA-256，避免公开原始邮箱 |
| content | text | not null | 评论纯文本 |
| status | varchar(16) | not null, indexed | `pending`、`approved`、`rejected`、`deleted` |
| ip_hash | char(64) | nullable | IP hash，用于限流和审计 |
| user_agent_hash | char(64) | nullable | User-Agent hash |
| created_at | timestamptz | not null | 创建时间 |
| updated_at | timestamptz | not null | 更新时间 |

索引建议：

```sql
create index comments_post_status_created_idx on comments (post_slug, status, created_at desc);
create index comments_status_created_idx on comments (status, created_at desc);
create index comments_parent_idx on comments (parent_id);
```

### 5.2 comment_audit_logs

第二阶段建议增加审计日志：

| 字段 | 类型 | 说明 |
|---|---|---|
| id | uuid | 日志 ID |
| comment_id | uuid | 评论 ID |
| action | text | `approve`、`reject`、`delete`、`restore` |
| actor | text | 操作者，第一版可固定为 `admin` |
| created_at | timestamptz | 操作时间 |

## 6. API 设计

### 6.1 公开接口

#### GET `/api/comments?postSlug=xxx`

返回指定评论目标下已审核内容。文章继续使用内容集合中的文章 slug；留言板使用保留标识 `__guestbook__`。

响应：

```json
{
  "comments": [
    {
      "id": "uuid",
      "postSlug": "2026/Manacher算法/index.md",
      "parentId": null,
      "authorName": "Abel",
      "content": "评论内容",
      "createdAt": "2026-07-09T15:00:00.000Z"
    }
  ]
}
```

#### POST `/api/comments`

提交评论，默认进入 `pending`。

请求：

```json
{
  "postSlug": "2026/Manacher算法/index.md",
  "parentId": null,
  "authorName": "Abel",
  "authorEmail": "name@example.com",
  "content": "评论内容",
  "turnstileToken": "token"
}
```

响应：

```json
{
  "status": "pending",
  "message": "评论已提交，审核后展示"
}
```

### 6.2 管理接口

#### POST `/api/admin/login`

使用管理员密码登录，返回短期 token 或设置 httpOnly cookie。

#### GET `/api/admin/comments?status=pending`

查看评论列表，默认按创建时间倒序。

#### PATCH `/api/admin/comments/:id/status`

更新状态。

请求：

```json
{
  "status": "approved"
}
```

#### DELETE `/api/admin/comments/:id`

软删除评论，实际将状态改为 `deleted`。

## 7. 安全策略

- 评论内容按纯文本处理，前端禁止使用 `dangerouslySetInnerHTML` 渲染评论。
- 后端限制字段长度：昵称 1 到 64，邮箱最大 254，正文 1 到 2000。
- 使用 Turnstile 校验提交请求。
- 按 IP hash、文章 slug 和邮箱 hash 做基础限流。
- CORS 仅允许博客正式域名和本地开发域名。
- 管理接口必须鉴权，token 设置过期时间。
- 邮箱和 IP 不存原文，使用带 salt 的 hash。
- `COMMENT_HASH_SALT` 必须使用生产环境独立随机值，避免邮箱、IP 和 User-Agent hash 被字典反查。
- 默认审核后公开，避免垃圾评论直接出现在页面。

## 7.1 留言板

- 前端路由为 `/guestbook`，通过复用评论 API 实现，不新增表或接口。
- 留言板固定使用 `postSlug=__guestbook__`。该值是内部稳定资源标识，不能替换为 `/guestbook` 或 `/about`，避免页面路由变更导致历史留言丢失。
- 留言板与文章评论按 `post_slug` 隔离，仍适用相同的审核、Turnstile、限流与纯文本渲染策略。
- 管理后台将 `__guestbook__` 显示为“留言板”，不向管理员暴露内部标识。

## 9. 前端接入点

文章详情页当前链路：

```txt
src/pages/blog/[...slug].astro
  -> src/layouts/BlogPost.astro
      -> <slot />
```

推荐在 `BlogPost.astro` 的正文 `<slot />` 后插入评论组件，并将 `post.id` 或规范化后的 slug 传入组件。当前 `BlogPost.astro` 只接收 frontmatter 数据，实施时需要从 `src/pages/blog/[...slug].astro` 把 `post.id` 一并传给布局。

同时建议在 `src/content.config.ts` 增加：

```ts
comments: z.boolean().optional().default(true),
```

单篇文章可通过 frontmatter 关闭评论：

```yaml
comments: false
```

## 10. 验证标准

- `pnpm build` 通过。
- 评论列表加载、空状态、错误状态、提交成功状态均可见。
- 未审核评论不会出现在公开文章页。
- 管理员可以批准、拒绝和删除评论。
- Turnstile 校验失败时不能提交。
- 评论正文中的 HTML 不会被浏览器执行。
- CORS 不允许未知 Origin 调用写接口。
